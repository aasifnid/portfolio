# Portfolio Curation Guide

Choosing which projects to include is itself a design decision. Four strong case studies beat eight mediocre ones — always. The portfolio reviewer has limited time. Density of evidence matters more than breadth of projects.

---

## What the Portfolio Must Show (As a Set)

Across all included case studies, the portfolio needs to demonstrate:

| Signal | At least 1 case study showing |
|--------|-------------------------------|
| Research-led design | User interviews, usability testing, analytics-driven decisions |
| Complex interaction or multi-screen flow | A problem that required thinking across multiple screens/states |
| Measurable outcome | A metric, even a rough one, that changed because of the work |
| Strategic thinking | A decision that involved trade-offs, scope, or direction — not just UI polish |
| Breadth of problem types | Not all the same kind of problem (don't show 4 onboarding flows) |

If one case study covers multiple signals, that's fine. The goal is portfolio-level coverage, not one-case-study-per-signal.

---

## Selection Criteria Per Project

Score each project on 4 factors (1 weak / 2 okay / 3 strong):

| Factor | What to assess |
|--------|---------------|
| **Ownership** | Can you say "I" for the core design work? |
| **Process visibility** | Do you have artifacts showing thinking — not just final screens? |
| **Outcome strength** | Did it ship? Was there a measurable or qualitative change? |
| **Problem interest** | Would a Google reviewer find this problem non-obvious or revealing? |

---

## How to Rank Projects for Aasif's Portfolio

When Aasif lists his projects, Folio scores each on:

```
Score = (Ownership clarity) + (Process visibility) + (Outcome strength) + (Problem interest)
Each factor: 1 (weak) / 2 (okay) / 3 (strong)
Max score: 12
```

**Include:** Score 9–12
**Consider:** Score 6–8 — include only if it fills a gap in the portfolio set
**Drop:** Score 1–5 — not portfolio-ready

## What to Do with Work That's Visually Strong but Story-Weak

Sometimes a project has beautiful final screens but weak process evidence — no research, no decision rationale, final screens only.

Options:
1. **Write it honestly** — "This project had no formal research. I made decisions based on..." and then show the design quality strongly
2. **Pair it with a different project** — use a process-rich but visually simpler project to balance it
3. **Don't include it** — if you can't tell a story about it, it's not a case study

**Never fabricate process.** "I ran user interviews" when you didn't is a lie that falls apart in interviews.

---

## Recommended Portfolio Shape for Google Pay Application

Given Aasif's background, aim for:

1. **Lead case study** — Most complex, most visible ownership, clearest outcome. This is the one a reviewer reads fully. Should show research + decision-making + shipped outcome.

2. **Process case study** — Strongest example of research and discovery. Could be a smaller feature with unusually good process evidence.

3. **Scale / systems case study** — Something that shows design thinking at a product level, not just screen level. A component system, a design pattern that standardised across the product, or a flow that touched multiple user types.

4. *(Optional)* **India-specific case study** — If you have a project specifically shaped by India context (NBU users, Tier 2/3 cities, device constraints, trust design), make it explicit. This is directly relevant to Google Pay's user base.

5. *(Optional)* **Exploratory or speculative** — If you've worked on something interesting that didn't ship, label it clearly as exploratory.
